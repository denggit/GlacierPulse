# V6.3.0-preflight: Semantic Restructure Plan

## 1. Current Reality

Current V6.2.9.5 is not a complete Fabio TripleA implementation. The live runtime is closer to:

- A1 iceberg absorption research
- A1 frozen zone lifecycle
- A1 reaction evaluator
- execution research candidate
- virtual position simulation
- outcome research

The names Phase1, Phase2, and Phase3 remain in the runtime code for compatibility, but their current meanings do not map cleanly to Fabio-style A1/A2/A3.

## 2. Current Runtime Chain

```text
OKX trades/books
-> MarketContext
-> Phase1Engine
-> IcebergDetector
-> IcebergZoneTracker
-> Phase2OrderflowEvaluator
-> Phase3CandidateEvaluator
-> VirtualPositionManager
-> Phase3OutcomeEvaluator
```

## 7. V6.3.1 Schema Adapter Direction

- V6.3.1 adds read-only A1 schema dataclasses and adapters.
- These adapters normalize old runtime dicts into A1 semantic names.
- Existing runtime dicts and legacy fields remain unchanged.
- The adapters do not participate in live trading decisions yet.
- A1OutcomeRecord normalizes IcebergOutcomeEvaluator fields (`mfe` / `mae`) and Phase3 virtual outcome fields (`max_favorable_u` / `max_adverse_u`) into `mfe_u` / `mae_u`.

## 8. V6.3.2 A1 Reaction Implementation Migration

- V6.3.2 moves the A1 reaction evaluator implementation to `src/strategy/a1_reaction/reaction_evaluator.py`.
- The legacy path `src/strategy/phase2_orderflow_evaluator.py` remains as a deprecated compatibility wrapper.
- Runtime behavior, PHASE2 log tags, state names, and `phase2_type` fields remain unchanged.
- This is the first step toward making the V6.3 semantic architecture the primary implementation path.
- No A2 or A3 logic is implemented in this version.

## 9. V6.3.3 Execution Research Candidate Implementation Migration

- V6.3.3 moves the execution research candidate/risk-gate implementation to `src/strategy/execution_research/candidate_evaluator.py`.
- The legacy path `src/strategy/phase3_candidate_evaluator.py` remains as a deprecated compatibility wrapper.
- Runtime behavior, PHASE3 log tags, candidate decisions, risk sizing, `phase2_type`, and `candidate_type` fields remain unchanged.
- This migration prevents the current execution research layer from being confused with future Fabio-style A3 aggression/ignition.
- No A2 or A3 logic is implemented in this version.

## 10. V6.3.4 Phase1Engine Runtime Import Migration

- V6.3.4 updates `Phase1Engine` to import A1 reaction and execution research candidate classes from semantic paths.
- `A1ReactionEvaluator` is imported from `src/strategy/a1_reaction/reaction_evaluator.py`.
- `ExecutionResearchCandidateEvaluator` is imported from `src/strategy/execution_research/candidate_evaluator.py`.
- Existing instance attribute names such as `phase2_orderflow_evaluator` and `phase3_candidate_evaluator` remain unchanged for compatibility.
- Legacy wrapper modules remain available for older imports.
- Runtime behavior, PHASE2/PHASE3 log tags, state names, `phase2_type`, `candidate_type`, candidate decisions, risk sizing, and virtual position behavior remain unchanged.
- No A2 or A3 logic is implemented in this version.


## 11. V6.3.5 A1 Reaction to Virtual Position Chain Switch

- V6.3.5 adds `A1_REACTION_TO_VIRTUAL_POSITION_ENABLED`.
- The default is `False`.
- A1 reaction confirmed events still flow into execution research candidate evaluation.
- Execution research candidate results are still generated and logged.
- By default, candidate results are no longer sent directly to `VirtualPositionManager.on_candidate`.
- Setting `A1_REACTION_TO_VIRTUAL_POSITION_ENABLED=True` restores the old virtual-position simulation path.
- `VIRTUAL_POSITION_MANAGER_ENABLED` still controls whether the virtual manager is created.
- `A1_REACTION_TO_VIRTUAL_POSITION_ENABLED` controls whether A1 reaction candidates are allowed to enter the virtual manager.
- Runtime A1 reaction logic, candidate decision logic, risk sizing, PHASE2/PHASE3 log tags, `phase2_type`, and `candidate_type` remain unchanged.
- No real trading is enabled.
- No A2 or A3 logic is implemented.


## 12. V6.3.6 A1 Unified Score Normalization

- V6.3.6 adds a research-only A1 unified score normalization model.
- The model combines `A1AbsorptionContext` and `A1ReactionSnapshot`.
- It outputs `a1_score`, `a1_quality_bucket`, `score_version`, and component breakdown.
- The score model is deterministic and pure: no network access, no trader access, no virtual-position access, and no runtime side effects.
- It does not replace `phase2_total_score` or `reaction_total_score`.
- It does not participate in candidate decisions, virtual position decisions, or order execution.
- Existing PHASE2/PHASE3 log tags, `phase2_type`, `candidate_type`, risk sizing, and virtual position behavior remain unchanged.
- No real trading is enabled.
- No A2 or A3 logic is implemented.


## 13. V6.3.7 A1 Research Report Aggregation

- V6.3.7 adds a research-only A1 report aggregation model.
- The report consumes `A1ScoreRecord` and `A1OutcomeRecord` through `A1ResearchSample`.
- It aggregates global stats and grouped stats by A1 quality bucket, reaction type, frozen reason, and candidate type.
- It reports sample count, win rate, average realized R, median realized R, profit factor, MFE, MAE, and best/worst R.
- The report is deterministic and pure: no network access, no trader access, no virtual-position access, no runtime side effects, and no log-file parsing.
- It does not replace `phase2_total_score`, `reaction_total_score`, `a1_score`, or existing outcome evaluator summaries.
- It does not participate in candidate decisions, virtual position decisions, or order execution.
- Existing PHASE2/PHASE3 log tags, `phase2_type`, `candidate_type`, risk sizing, and virtual position behavior remain unchanged.
- No real trading is enabled.
- No A2 or A3 logic is implemented.
