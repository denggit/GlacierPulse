# V6.3.10 A1 Edge Validation Suite

## Goal

V6.3.10 is an offline research toolkit. It does not implement A2 Accumulation, A3 Aggression, real trading, virtual positions, or any runtime behavior change.

A1 Edge Validation Suite is offline-only. It is intentionally not imported by `main.py`. `main.py` only collects A1 reaction events, usually into `logs/research/a1_reaction_events.jsonl`; `tools/analyze_a1_edge.py` runs later against local files.

The goal is to answer one question before V6.4 A2 design: does A1 have statistical edge?

The suite checks whether A1 events have stronger future MFE than random points, stronger future volatility than random points, directional prediction power, useful subtypes, useless subtypes, and entry hypotheses worth carrying into A2.

## Why This Exists

A2 should not be built on top of an unvalidated A1 signal. A1 may be a directional signal, a volatility warning, a mixed subtype family, or noise. V6.3.10 separates these cases with stable offline exports and reproducible random baselines before any new runtime trading logic is added.

## Input Data

A1 events can come from:

- `logs/research/a1_reaction_events.jsonl`
- `logs/research/a1_edge_events.csv`
- a user supplied JSONL or CSV file with compatible fields

The recorder JSONL produced by `A1ReactionEventRecorder` is supported directly.

`event_key` is the offline analysis primary key. Do not use `zone_id` alone as a unique event key because one zone can emit multiple reaction events, and some research rows may not have a zone. If an input record already has `event_key`, it is preserved. Otherwise the exporter builds one from zone, reaction type, event kind, timestamp, direction, and zone bounds.

1m K line CSV must contain:

```csv
timestamp,open,high,low,close,volume
```

The timestamp column can also be named `ts`, `datetime`, or `time`. Timestamps may be seconds, milliseconds, or ISO strings. If required K line columns are missing, the CLI exits with a clear error.

## CLI Usage

```bash
python tools/analyze_a1_edge.py \
  --events logs/research/a1_reaction_events.jsonl \
  --klines data/ETH-USDT-SWAP_1m.csv \
  --out reports/a1_edge \
  --symbol ETH-USDT-SWAP \
  --random-samples-per-event 10 \
  --seed 42
```

Optional controls:

- `--windows 1m,3m,5m,15m,30m,60m`
- `--min-risk-u 1.0`
- `--min-risk-pct 0.0003`
- `--stop-buffer-u 0.8`
- `--roundtrip-fee-pct 0.001`
- `--min-group-sample-size 30`
- `--exclude-near-a1-minutes 5`
- `--baseline-risk-mode source`

The CLI is offline only. It does not import `main.py`, does not connect to OKX, does not read API keys, does not start websockets, and does not call trader or virtual position modules.

Correct workflow:

```text
main.py shadow run -> generate A1 reaction JSONL
tools/analyze_a1_edge.py -> offline analysis
```

The suite will not run automatically inside `main.py`.

## Output Files

Default output directory:

```text
reports/a1_edge/
```

Files:

- `a1_edge_events.csv`: normalized A1 event dataset with stable columns
- `a1_edge_events.jsonl`: normalized event JSONL
- `a1_edge_dataset_summary.json`: dataset export summary before report building
- `a1_edge_summary.json`: final report summary with decision
- `a1_forward_metrics.csv`: A1 future MFE/MAE/range/hit metrics by window
- `a1_random_baseline.csv`: same-window random baseline metrics
- `a1_vs_random_summary.csv`: grouped A1 vs random comparison
- `a1_vs_random_summary.md`: markdown summary of A1 vs random
- `a1_hypothesis_results.csv`: per-event entry hypothesis rows
- `a1_hypothesis_summary.csv`: grouped hypothesis statistics
- `a1_best_hypothesis_by_group.csv`: hypothesis groups sorted by proxy realized R
- `a1_go_no_go_report.md`: final decision report

Existing files in the output directory may be overwritten.

## A1 vs Random Interpretation

Random events are sampled from the same day as each A1 event, use the same direction, avoid the A1 timestamp neighborhood, and are reproducible with `--seed`.

Baseline risk mode:

- `source`: default and recommended. Every random sample uses its source A1 event's `source_risk_u` as the R denominator. This compares A1 movement / A1 risk against random movement / the same A1 risk.
- `local`: uses a local random-point proxy width derived from `source_risk_u` around the random entry price. It does not compare the random entry against the original A1 zone bounds.

This avoids the incorrect case where a random entry far away from the A1 frozen zone gets an artificially huge risk denominator.

Read `a1_vs_random_summary.csv`:

- `STRONG_DIRECTIONAL_EDGE`: A1 has stronger 15m MFE and +1R hit rate than random with sufficient sample size.
- `VOLATILITY_EDGE`: A1 expands range more than random, but direction is not clearly better.
- `NO_EDGE`: A1 is not materially better than random.
- `INSUFFICIENT_SAMPLE`: sample size is too small for a conclusion.

## Hypothesis Simulator Interpretation

The simulator is a research proxy, not a backtest and not a trading engine. It does not call virtual position manager or trader.

Supported first version hypotheses include:

- `IMMEDIATE_ENTRY`
- `NEXT_BAR_ENTRY`
- `DELAYED_ENTRY_1M`
- `DELAYED_ENTRY_3M`
- `FAST_CLEAN_HOLD_ENTRY`
- `ZONE_RETEST_ENTRY`
- `BREAKOUT_AWAY_ENTRY`
- `FAILED_RECLAIM_REVERSE_ENTRY`

If a hypothesis condition is not met, the result is emitted as a skipped row with `skip_reason`. Focus on groups with sufficient `valid_count`, positive `avg_realized_r_proxy`, controlled `hit_stop_rate`, and a meaningful right tail.

## Go / No-Go Decision

`a1_go_no_go_report.md` ends with one decision:

- `A1_GO`: proceed to V6.4 A2 design using the strongest A1 groups.
- `A1_PARTIAL_GO`: proceed to V6.4 only for selected A1 subtypes and exclude weak/no-edge groups.
- `A1_NO_GO`: do not proceed to A2; rework A1 detection.
- `INSUFFICIENT_SAMPLE`: continue data collection and do not make a strategic conclusion.

## Recommended Run Flow

### Step 1: 2-4 Hour Smoke Run

```bash
V62_LOG_PROFILE=RESEARCH_KEY_EVENTS \
LOG_TO_CONSOLE=false \
LOG_TO_FILE=true \
A1_REACTION_RESEARCH_COVERAGE_ENABLED=true \
A1_REACTION_EVENT_RECORDER_ENABLED=true \
A1_REACTION_EVENT_RECORDER_WRITE_JSONL=true \
A1_REACTION_TO_VIRTUAL_POSITION_ENABLED=false \
REAL_EXECUTION_ENABLED=false \
PHASE3_REAL_TRADING_ENABLED=false \
python main.py
```

This verifies JSONL recording and K line availability. Real trading remains disabled.

`main.py` does not import or run `src/research/a1_edge`. This is intentional: the validation suite is a post-run research step, not a runtime component.

### Step 2: Run Analysis

```bash
python tools/analyze_a1_edge.py \
  --events logs/research/a1_reaction_events.jsonl \
  --klines data/ETH-USDT-SWAP_1m.csv \
  --out reports/a1_edge \
  --symbol ETH-USDT-SWAP \
  --random-samples-per-event 10 \
  --seed 42
```

### Step 3: Check Report

```text
reports/a1_edge/a1_go_no_go_report.md
```

### Step 4: 24 Hour Shadow Run

Repeat the smoke run for a full 24 hours with:

```bash
A1_REACTION_TO_VIRTUAL_POSITION_ENABLED=false \
REAL_EXECUTION_ENABLED=false \
PHASE3_REAL_TRADING_ENABLED=false
```

Then rerun the CLI and compare A1 vs random plus hypothesis summaries.

### Step 5: 3-7 Day Validation

Collect multiple market sessions and rerun the suite daily. Do not proceed to A2 from one thin sample. Look for stable edge labels by A1 subtype, frozen reason, frozen state, and direction.

## Safety Notes

The default safety posture must remain:

- `A1_REACTION_TO_VIRTUAL_POSITION_ENABLED=false`
- `REAL_EXECUTION_ENABLED=false`
- `PHASE3_REAL_TRADING_ENABLED=false`

V6.3.10 is research data export, random baseline, forward metrics, entry hypothesis simulation, and report generation only.
