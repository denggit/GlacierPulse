# V6.3.10.3 A1 Edge Timezone Guard

## What Changed

`a1_reaction_events.jsonl` keeps `reaction_event_ts` as UTC epoch seconds. This is the correct event-time source and is not changed by V6.3.10.3.

Runtime and nohup logs usually show machine local time, for example Beijing time. A log line like `2026-05-22 00:23:38` can align with `reaction_event_ts=1779380618.267`; the JSONL is not wrong just because it is epoch seconds.

The ambiguity was in 1m K-line CSV files where `timestamp` looked like `2026-05-21 20:00:00`. Those strings were Beijing local time but had no `+08:00`, `Asia/Shanghai`, or epoch field. The A1 edge analyzer now treats legacy naïve K-line timestamps as `Asia/Shanghai` by default, and the CLI exposes the timezone explicitly.

## Recommended Export

Use:

```bash
python tools/export_history_k.py \
  --symbol ETH-USDT-SWAP \
  --timeframe 1m \
  --start "2026-05-21 20:00" \
  --end "2026-05-22 20:00" \
  --timezone Asia/Shanghai
```

The exported CSV keeps the old `timestamp` field and adds:

- `timestamp_epoch_sec`
- `timestamp_ms`
- `timestamp_utc`
- `timestamp_local`
- `timezone`

Prefer `timestamp_epoch_sec` for offline analysis because it removes timezone ambiguity.

## Correct Analysis Command

```bash
python tools/analyze_a1_edge.py \
  --events logs/research/a1_reaction_events.jsonl \
  --klines data/ETH-USDT-SWAP_1m.csv \
  --out reports/a1_edge \
  --symbol ETH-USDT-SWAP \
  --random-samples-per-event 10 \
  --seed 42 \
  --kline-timezone Asia/Shanghai
```

If the K-line CSV already has `timestamp_epoch_sec` or `timestamp_ms`, those fields are used before any string timezone parsing.

## Coverage Guard

Forward metrics now mark rows as `insufficient_future_data=true` when future bars cover less than 60% of the expected window. For 1m K lines, 15m expects about 15 bars and 60m expects about 60 bars.

Rows marked `insufficient_future_data=true` are excluded from A1 vs random core statistics, random baseline comparison, and hypothesis valid sample counts. If most events lack future K-line coverage, the report decision becomes `INSUFFICIENT_KLINE_COVERAGE`.
